# Break Reminder (Android)

A tiny, lightweight Flutter app that reminds you to take a break — on your schedule.

Press **Start work** on the home screen to anchor the cycle: one interval later (default 1 hour), a reminder fires as a **constantly buzzing** notification (insistent vibration that doesn't stop until you act) with **Take break** / **Skip** actions:

- **Take break** opens the app straight into a stopwatch that records the break; the screen stays awake while it's visible. When you're done, tap **End break** — the break (start, end, duration) is logged to a local SQLite database and the next reminder is scheduled exactly one interval after the break ended.
- **Skip** logs a skipped break without opening the app and restarts the timer — the next reminder lands one interval after the skip.

While a session is running the Start work button becomes **End work**. Pressing it closes the day: all reminders stop and a summary dialog reports when you started and ended, time worked, breaks taken (count and minutes), breaks skipped, and net focus time.

Reminders only fire inside your **active schedule**: pick which weekdays it runs (e.g. only Monday and Wednesday) and a daily time window (e.g. 9:00 AM – 6:00 PM). A reminder that would land outside the window rolls forward to the next active day's window start.

The home screen is a control panel: stat tiles (breaks today, break time today/this week/all-time), a week/month/year trend chart of break minutes, and a GitHub-style year heatmap with streak / best-day / year-total callouts.

## How scheduling works (design note)
Because the cycle re-anchors whenever the user acts (Start work, Skip, End break), a single repeating notification can't model it. The app keeps a **chain of one-shot scheduled notifications** (the next ~24 occurrences, computed inside the active window, delivered as exact alarms when permitted):
- Start work and End break happen in the UI isolate: they cancel the chain and reschedule it anchored at that instant. End work cancels the chain and clears the anchor, so nothing fires until the next Start work.
- Skip fires in a headless isolate where the notifications plugin can't reschedule, so it only logs the skip; the app re-anchors at the skip instant the next time it runs. Until then the chain's own cadence (each link one interval after the previous) already approximates it, since the insistent buzz makes the user act within seconds.
- No reminders are scheduled until Start work has been pressed at least once.
- The chain is topped back up on app launch when it runs low, so it can't run dry in normal use. Scheduled notifications survive reboots (handled by `flutter_local_notifications`' boot receiver).

## Project layout
- `lib/core/db/` — SQLite layer (`sqlite3` via FFI, safe in the background notification isolate). Table: `breaks(id, start_ts, end_ts, duration_seconds, status)` with status `completed` or `skipped`.
- `lib/core/schedule/` — pure schedule math (active days + daily window + interval → next reminder instants).
- `lib/core/notifications/` — scheduling chain + Take break / Skip handling, including while the app isn't running.
- `lib/core/settings/` — pause/resume, interval, active days, active hours, cycle anchor, in-progress break persistence.
- `lib/features/home/` — control panel (Start work button, next-break readout, stat tiles, trend chart, heatmap).
- `lib/features/break/` — the stopwatch screen (keeps the display awake via `wakelock_plus`).
- `lib/features/settings/` — pause toggle, interval stepper, weekday chips, time-window pickers, test reminder.

The Android-native scaffold (`android/`) isn't committed — it should be generated by the Flutter tool itself so it matches your installed Flutter version (step 2 below).

## Setup

### 1. Install Flutter and Android tooling
1. Install the **Flutter SDK** (stable channel) and add it to your PATH: https://docs.flutter.dev/get-started/install/windows
2. Install **Android Studio** (easiest way to get the Android SDK, `adb`, and license acceptance): https://developer.android.com/studio
3. Run `flutter doctor` and resolve anything it flags (`flutter doctor --android-licenses` if prompted).

### 2. Generate the Android project scaffold
From the repo root:
```powershell
flutter create --platforms=android --org com.jaichand .
```
This adds the `android/` folder without touching the `lib/` code. It derives the app id `com.jaichand.break_reminder_app` from the `--org` plus `name: break_reminder_app` in pubspec.yaml — this must match the `kApplicationId` constant in `lib/core/db/database.dart` (it already does; don't change one without the other).

Afterwards, open `android/app/src/main/AndroidManifest.xml` and set:
```xml
<application android:label="Break Reminder" ...>
```

### 3. Get dependencies and run
```powershell
flutter pub get
flutter run
```
A real phone (USB debugging) is recommended — background/notification behavior only shows up reliably on a real device.

## Verification checklist
1. App launches to the home screen showing all-zero stats and "Press Start work to begin the hourly break cycle."
2. Tap **Start work** → "Next break at …" shows one interval from now.
3. Settings → **Send test reminder now** → a heads-up notification appears with Take break / Skip actions and buzzes continuously until you act on it.
4. Tap **Take break** → app opens into the stopwatch and the screen stays awake; wait a bit, tap **End break** → home shows breaks today = 1, the time you took, and the next break exactly one interval after you ended.
5. Send another test reminder, tap **Skip** → nothing appears in stats (skips are logged but not counted) and the next break re-anchors to one interval after the skip.
6. Kill the app mid-break, reopen → the stopwatch resumes from the original start time.
7. Set active days/hours to a window that excludes right now, end a break → no reminder should fire until the window opens again.
8. Toggle **Pause** → no reminders; press Start work → they're back.
9. Over the following days: reminders fire only on the selected days inside the selected hours, and survive a reboot.

## The data
Breaks are logged to a local SQLite database in the app's private storage — nothing leaves the device. Uninstalling the app deletes it.
